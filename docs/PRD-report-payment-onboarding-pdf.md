## 1. 개요

LIFT는 퇴직, 이직 등 생애전환 상황에서 사용자가 놓치기 쉬운 행정 절차, 예상 수령액, 필요 서류, 공식 신청 링크를 로드맵 형태로 제공하는 서비스다.

이번 범위는 기존 단일 가격 결제와 단순 미리보기 구조를 확장해, 사용자가 결제 전에 자신이 실제로 무엇을 신청하거나 확인할 수 있는지 먼저 확인하고, 선택 가능한 리포트 플랜을 명확히 비교한 뒤 결제하도록 만드는 것이다. 또한 온보딩 카테고리 개편, 퇴직 사유 선택지 개편, 다음 일자리 확정일 입력, 모바일 PDF 저장 안정화까지 포함한다.

핵심 문제는 다음과 같다.

- 사용자가 결제했는데 신청 가능하거나 확인할 수 있는 항목이 없으면 안 된다.
- 기존 결제 가격이 4,900원 단일 구조라 플랜 구분이 없다.
- 온보딩에 현재 제공하지 않을 생애 이벤트가 활성 선택지처럼 보인다.
- 퇴직 사유 선택지에 제거해야 할 항목이 있고, 새 사유가 필요하다.
- 다음 일자리가 확정된 사용자는 시작일을 입력할 수 있어야 한다.
- 모바일에서 PDF 저장이 브라우저 인쇄 방식 때문에 불안정하다.
- PDF 저장 모달 UX가 복잡하고, 특히 월급 입력 기반 PDF 저장 경로가 명확하지 않다.

## 2. 목표

### 2.1 제품 목표

1. 결제 전에 사용자가 신청 가능하거나 확인할 수 있는 항목을 명확히 보여준다.
2. 결제 플랜을 `기본 리포트 6,900원`, `확장 리포트 13,900원` 두 가지로 분리한다.
3. 사용자가 선택한 플랜 가격이 결제 금액에 정확히 반영되도록 한다.
4. 신청 가능 항목이 없는 사용자가 결제하지 못하게 방지한다.
5. 온보딩 첫 화면에서 현재 지원하는 이벤트와 추후 지원 예정 이벤트를 명확히 구분한다.
6. 퇴직 사유 입력에서 `권고사직`을 사용자 선택지에서 제거하고, `정년퇴직`을 추가한다.
7. 다음 일자리 상태가 `확정됨`이면 시작일 달력을 제공한다.
8. 모바일에서도 PDF 저장 흐름이 안정적으로 작동하도록 전용 PDF 화면을 제공한다.
9. PDF 저장 시 월급 입력 여부에 따른 계산 결과 차이를 명확히 한다.

### 2.2 사용자 목표

1. 내가 결제하기 전에 무엇을 신청할 수 있는지 확인하고 싶다.
2. 리포트 플랜별 가격과 제공 내용을 비교하고 선택하고 싶다.
3. 내가 해당하지 않는 카테고리를 실수로 선택하지 않고 싶다.
4. 현재는 못 쓰는 기능도 앞으로 제공될 예정인지 알 수 있으면 좋겠다.
5. 다음 직장이 정해졌다면 시작일을 입력해 더 정확한 로드맵을 만들고 싶다.
6. PDF 저장 시 월급을 입력하면 더 구체적인 금액이 계산되는지 알고 싶다.
7. 핸드폰에서도 PDF 저장 버튼을 눌렀을 때 저장/공유 가능한 화면이 안정적으로 열리면 좋겠다.

## 3. 비목표

이번 범위에서 다루지 않는 것:

1. 결혼/출산휴가 로드맵의 실제 분석 로직 구현
2. 결혼/출산휴가 결제 또는 신청 항목 제공
3. 플랜별 권한 차이를 서버 영속 데이터에 완전히 저장하는 과금 시스템 구현
4. 실제 운영 PG 결제 심사/정산/환불 시스템
5. PDF 파일을 서버에서 바이너리로 생성해 직접 다운로드하는 기능
6. 자동 서류 조회의 실제 공공 마이데이터 연동
7. 권고사직 enum/기존 데이터 완전 삭제

## 5. 사용자 여정

### 5.1 신규 진단 여정

1. 사용자가 로그인한다.
2. 온보딩 상황 선택 화면에 진입한다.
3. 사용자는 `퇴직` 또는 `이직` 중 하나를 선택할 수 있다.
4. `결혼`, `출산휴가`는 회색 잠금 카드로 보이지만 선택할 수 없다.
5. 사용자는 약관에 동의하고 다음 단계로 이동한다.
6. 진단 입력 화면에서 퇴사일/마지막 근무일, 퇴사 사유, 다음 일자리 상태, 고용보험 기간, 현재 소득 상태 등을 입력한다.
7. 다음 일자리 상태가 `확정됨`이면 `다음 일자리 시작일` 달력 필드가 추가로 표시된다.
8. 사용자가 로드맵을 생성하면 무료 미리보기 페이지로 이동한다.

### 5.2 결제 여정

1. 무료 미리보기에서 사용자는 전체 항목 수와 상위 항목 1~2개를 확인한다.
2. `상세 리포트 결제하고 전부 보기` 버튼을 누른다.
3. 결제 화면 진입 시 시스템은 리포트 미리보기 API를 다시 조회한다.
4. 결제 화면 상단에는 `지금 신청 검토할 수 있는 항목` 카드가 표시된다.
5. 신청 가능성이 높거나 확인이 필요한 항목 수가 1개 이상이면 결제 버튼이 활성화된다.
6. 사용자는 `기본 리포트 6,900원` 또는 `확장 리포트 13,900원` 중 하나를 선택한다.
7. 선택한 플랜 이름과 가격이 결제 금액 영역에 반영된다.
8. 사용자는 데모 결제 또는 토스 테스트 결제를 진행한다.
9. 결제가 완료되면 상세 리포트로 이동한다.

### 5.3 신청 가능 항목 없음 방지 여정

1. 사용자가 결제 화면에 진입한다.
2. 시스템은 `actionableItemCount`를 확인한다.
3. `actionableItemCount`가 0이면 안내 문구를 표시한다.
4. 데모 결제와 토스 결제 버튼을 비활성화한다.
5. 사용자는 이전 화면으로 돌아가 입력 정보를 수정할 수 있다.

### 5.4 PDF 저장 여정

1. 사용자가 상세 리포트 화면에서 `예상 수령액 리포트 PDF로 저장` 버튼을 누른다.
2. PDF 저장 모달이 열린다.
3. 사용자는 월 평균임금을 입력할 수 있다.
4. `월급 입력해서 저장`을 누르면 월급 기반 금액 계산이 적용된 PDF 전용 화면으로 이동한다.
5. `월급 없이 저장`을 누르면 범위/산식 중심의 PDF 전용 화면으로 이동한다.
6. PDF 전용 화면에는 상단에 `이전으로`, `PDF 저장하기` 버튼이 표시된다.
7. 사용자가 `PDF 저장하기` 버튼을 직접 누르면 브라우저 인쇄/저장 UI가 열린다.
8. 모바일에서도 사용자 터치에 의해 인쇄 UI가 열리므로 자동 print보다 안정적이다.

## 6. 기능 요구사항

## 6.1 온보딩 생애 이벤트 선택

### FR-ONB-001: 선택 가능한 이벤트

온보딩 첫 화면에서 사용자가 선택 가능한 이벤트는 다음으로 제한한다.

- 퇴직
- 이직

### FR-ONB-002: 제거되는 이벤트

기존 선택 가능 이벤트였던 `실직`은 온보딩 첫 화면에서 제거한다.

주의:
- 내부 enum이나 기존 데이터 호환을 위해 `UNEMPLOYMENT` 타입 자체는 유지될 수 있다.
- 단, 신규 사용자가 온보딩 첫 화면에서 `실직`을 직접 선택할 수 있으면 안 된다.

### FR-ONB-003: 잠금 이벤트

온보딩 첫 화면에 다음 이벤트를 회색 잠금 카드로 표시한다.

- 결혼
- 출산휴가

각 카드는 다음 조건을 만족해야 한다.

- 버튼 또는 카드처럼 보이되 `disabled` 상태여야 한다.
- 클릭/터치해도 선택되지 않아야 한다.
- `준비 중` 라벨이 표시되어야 한다.
- 색상은 활성 카드보다 낮은 대비의 회색 톤이어야 한다.
- 추후 기능 예정이라는 느낌을 주되, 현재 이용 가능한 기능으로 오해되면 안 된다.

### FR-ONB-004: 약관 동의

기존처럼 약관 동의 체크가 완료되어야 다음 단계로 이동할 수 있다.

## 6.2 퇴사 사유 선택

### FR-REASON-001: 제거 항목

사용자 입력 UI에서 `권고사직` 선택지를 제거한다.

적용 위치:

- `/assessment/new`
- `/my` 새 로드맵 만들기 폼

주의:
- 서버 enum `RECOMMENDED_RESIGNATION`은 기존 저장 데이터와 API 호환을 위해 유지할 수 있다.
- 사용자가 신규 UI에서 해당 값을 선택할 수 있으면 안 된다.

### FR-REASON-002: 추가 항목

퇴사 사유 선택지에 `정년퇴직`을 추가한다.

내부 값:

- `MANDATORY_RETIREMENT`

표시명:

- `정년퇴직`

적용 위치:

- `/assessment/new`
- `/my` 새 로드맵 만들기 폼
- 공통 라벨 매핑
- 프론트 타입
- 백엔드 enum

### FR-REASON-003: 실업급여 판단

`정년퇴직`은 현재 실업급여 `HIGH` 판단 사유로 자동 분류하지 않는다.

현재 실업급여 신청 가능성이 높은 사유:

- 계약 만료
- 권고사직
- 회사 폐업

주의:
- `권고사직`은 UI에서는 제거되지만 기존 데이터 호환상 룰에는 남아 있을 수 있다.
- `정년퇴직`은 개별 상황 확인이 필요할 수 있으므로 기본적으로 `확인 필요` 또는 기타 룰 판단에 따른다.

## 6.3 다음 일자리 상태 및 날짜

### FR-JOB-001: 다음 일자리 상태

다음 일자리 상태 선택지는 기존처럼 유지한다.

- 아직 미정
- 확정됨
- 모름

### FR-JOB-002: 확정됨 선택 시 시작일 필드 노출

사용자가 `확정됨`을 선택하면 `다음 일자리 시작일` 날짜 필드가 표시되어야 한다.

적용 위치:

- `/assessment/new`
- `/my` 새 로드맵 만들기 폼

### FR-JOB-003: 시작일 달력

`다음 일자리 시작일`은 기존 `DateField` 컴포넌트를 사용한다.

지원 기능:

- 날짜 트리거 버튼
- 달력 팝오버
- 이전 달/다음 달 이동
- 오늘 선택
- 날짜 삭제
- 빠른 선택 칩

### FR-JOB-004: 상태 변경 시 날짜 초기화

사용자가 다음 일자리 상태를 `확정됨`에서 다른 값으로 바꾸면 `nextJobStartDate`는 빈 값으로 초기화한다.

### FR-JOB-005: API 요청 필드

진단 생성 요청에 다음 필드를 추가한다.

```json
{
  "nextJobStartDate": "2026-08-01"
}
```

값 규칙:

- `nextJobStatus === CONFIRMED`이고 날짜가 있으면 ISO `yyyy-mm-dd` 문자열 전송
- 그 외에는 `null` 전송

### FR-JOB-006: 백엔드 저장

백엔드 `LifeAssessment`는 `next_job_start_date` 컬럼을 가진다.

용도:

- 사용자 입력 보존
- 향후 룰 엔진, PDF, 공공 혜택 추천 정확도 개선에 활용 가능

## 6.4 결제 플랜

### FR-PAY-001: 플랜 구성

결제 플랜은 2개로 구성한다.

#### 기본 리포트

- 가격: 6,900원
- 설명: 신청 가능 항목, 우선순위, 필요 서류와 공식 링크 제공

#### 확장 리포트

- 가격: 13,900원
- 설명: 기본 리포트 + AI 질문 10회 + PDF 저장용 금액 계산

### FR-PAY-002: 플랜 선택 UI

결제 화면에서 두 플랜 카드를 나란히 보여준다.

요구사항:

- 두 카드의 높이는 항상 같아야 한다.
- 기본 리포트 설명이 짧더라도 확장 리포트보다 카드가 커지거나 작아지면 안 된다.
- 선택된 플랜은 시각적으로 강조한다.
- 모바일에서는 한 열로 쌓이되 각 카드의 최소 높이는 안정적으로 유지한다.

### FR-PAY-003: 기본 선택값

결제 화면 최초 진입 시 기본 선택 플랜은 `기본 리포트`다.

### FR-PAY-004: 결제 금액 표시

플랜 선택에 따라 하단 결제 금액 영역이 즉시 갱신되어야 한다.

예:

- 기본 리포트 선택 시 `결제 금액 6,900원`
- 확장 리포트 선택 시 `결제 금액 13,900원`

### FR-PAY-005: 토스 결제 금액

토스 테스트 결제 요청 시 선택된 플랜 가격을 `amount.value`로 전달한다.

허용 금액:

- 6,900
- 13,900

서버는 위 두 금액 외의 결제 승인 요청을 거부해야 한다.

### FR-PAY-006: 데모 결제

데모 결제는 현재처럼 리포트 결제 상태를 `PAID`로 전환한다.

주의:
- 현재 범위에서는 데모 결제 시 선택한 플랜을 서버에 저장하지 않는다.
- 추후 실제 과금/권한 분기 시 플랜 저장 필드가 필요하다.

## 6.5 결제 전 신청 가능 항목 확인

### FR-CHECKOUT-001: 미리보기 재조회

결제 화면 진입 시 리포트 미리보기 API를 다시 조회한다.

목적:

- 사용자가 결제 직전에 실제 열릴 항목을 확인
- 이미 결제된 리포트면 상세 리포트로 이동
- 신청 가능 항목이 없으면 결제 차단

### FR-CHECKOUT-002: 신청 검토 항목 카드

결제 화면 상단에 `결제 전 확인` 영역을 표시한다.

포함 정보:

- 제목: `지금 신청 검토할 수 있는 항목`
- 신청 검토 가능 항목 수
- 리포트 요약 메시지
- 하이라이트 항목 목록
- 각 항목의 신청 가능성 배지
- 각 항목의 우선순위 배지
- 숨겨진 항목 수 안내

### FR-CHECKOUT-003: actionableItemCount

서버 미리보기 응답에 `actionableItemCount`를 추가한다.

정의:

- `eligibilityLevel !== LOW`인 항목 수
- 즉, 신청 가능성이 높거나 확인이 필요한 항목 수

응답 예:

```json
{
  "reportId": 1,
  "totalItemCount": 3,
  "actionableItemCount": 2,
  "highlightItems": []
}
```

### FR-CHECKOUT-004: 결제 버튼 잠금

`actionableItemCount`가 0이면 결제 버튼을 비활성화한다.

비활성화 대상:

- 데모 결제 버튼
- 토스 테스트 결제 버튼

표시 문구:

- `입력하신 정보로는 현재 신청 가능성이 높거나 확인 필요한 항목이 없습니다. 결제 전에 정보를 다시 확인해 주세요.`

### FR-CHECKOUT-005: 결제 버튼 활성화

`actionableItemCount`가 1 이상이면 결제 버튼을 활성화한다.

## 6.6 상세 리포트 PDF 저장

### FR-PDF-001: 기존 방식 문제

기존 방식은 상세 리포트 페이지 안에 숨겨진 `.pdf-doc`을 렌더링하고 `window.print()`를 짧은 timeout 후 호출했다.

문제:

- 모바일 브라우저에서 자동 print가 사용자 제스처로 인정되지 않을 수 있다.
- 숨겨진 문서 렌더링 타이밍이 불안정하다.
- PDF 저장 모달 내 추천 카드가 실제 저장 액션이 아니라 입력 포커스만 유도해 UX가 혼란스럽다.
- 화면용 UI와 인쇄용 문서가 한 페이지에 섞여 있어 유지보수가 어렵다.

### FR-PDF-002: PDF 저장 전용 화면

PDF 저장은 전용 경로에서 처리한다.

경로:

```text
/report/[id]/pdf
/report/[id]/pdf?monthlyAverageWage=3000000
```

전용 화면 구성:

- 상단 툴바
  - 이전으로
  - PDF 저장하기
- PDF 문서 미리보기
- 인쇄 시 툴바 숨김

### FR-PDF-003: 월급 입력 PDF

사용자가 월 평균임금을 입력하고 `월급 입력해서 저장`을 누르면 다음 경로로 이동한다.

```text
/report/[id]/pdf?monthlyAverageWage={amount}
```

동작:

- 해당 금액을 `api.getPdfReport`에 전달
- 서버는 해당 금액으로 예상 수령액/절감액을 계산한 리포트 데이터를 반환
- PDF 전용 화면은 계산된 데이터를 표시

### FR-PDF-004: 월급 없이 PDF

사용자가 `월급 없이 저장`을 누르면 다음 경로로 이동한다.

```text
/report/[id]/pdf
```

동작:

- `monthlyAverageWage` 없이 PDF 리포트 데이터 요청
- 금액 범위와 산식 중심으로 PDF 표시

### FR-PDF-005: PDF 저장 버튼

PDF 전용 화면에서 `PDF 저장하기` 버튼을 누르면 즉시 `window.print()`를 호출한다.

목적:

- 모바일에서 사용자 터치 직후 print를 호출해 팝업/인쇄 차단 가능성을 낮춤
- 사용자가 직접 저장 동작을 수행한다고 인지할 수 있음

### FR-PDF-006: 모바일 레이아웃

모바일 폭에서 PDF 전용 화면은 다음을 만족해야 한다.

- `이전으로`, `PDF 저장하기` 버튼은 한 열로 표시
- 버튼 텍스트가 잘리지 않아야 함
- PDF 문서는 화면 너비에 맞게 표시
- 인쇄 시에는 배경/툴바 없이 PDF 문서만 남아야 함

### FR-PDF-007: PDF 모달 단순화

상세 리포트의 PDF 저장 모달은 다음 요소만 가진다.

- 제목: `PDF 저장 방식 선택`
- 설명: `월급 입력 여부에 따라 금액 표시가 달라져요`
- 월 평균임금 입력 필드
- `월급 입력해서 저장`
- `월급 없이 저장`
- 오류 메시지 영역

제거:

- 추천 카드
- 월급 없이 PDF 카드
- 카드 클릭 시 입력 포커스만 이동하는 흐름

### FR-PDF-008: 월급 입력 검증

`월급 입력해서 저장` 클릭 시 다음 검증을 수행한다.

- 숫자만 추출
- 0보다 큰 값이어야 함
- 값이 없거나 0 이하면 `월 평균임금을 입력해 주세요.` 표시

서버 검증:

- 최소 1원
- 최대 100,000,000원

## 11. QA 시나리오

### 11.1 온보딩

1. 온보딩 진입
2. `퇴직`, `이직`만 활성 선택 가능한지 확인
3. `실직` 카드가 없는지 확인
4. `결혼`, `출산휴가` 카드가 보이는지 확인
5. 두 카드가 disabled인지 확인
6. 잠금 카드를 눌러도 선택되지 않는지 확인

### 11.2 퇴사 사유

1. `/assessment/new` 진입
2. 퇴사 사유에 `권고사직`이 없는지 확인
3. `정년퇴직`이 있는지 확인
4. `/my` 새 로드맵 만들기에서도 동일한지 확인

### 11.3 다음 일자리 시작일

1. `/assessment/new`에서 다음 일자리 `아직 미정` 선택
2. `다음 일자리 시작일`이 보이지 않는지 확인
3. `확정됨` 선택
4. `다음 일자리 시작일`이 표시되는지 확인
5. 달력 팝오버가 열리는지 확인
6. 날짜 선택 후 값이 표시되는지 확인
7. 다시 `아직 미정` 선택
8. 시작일 필드가 사라지고 값이 초기화되는지 확인

### 11.4 `/my` 달력 잘림

1. `/my` 진입
2. 새 로드맵 만들기에서 다음 일자리 `확정됨` 선택
3. 시작일 달력 열기
4. 달력 팝오버가 카드 하단에 잘리지 않는지 확인

### 11.5 결제 플랜

1. 무료 미리보기에서 결제 화면 진입
2. 신청 검토 항목 카드가 상단에 보이는지 확인
3. 기본 리포트와 확장 리포트가 모두 보이는지 확인
4. 두 카드 높이가 같은지 확인
5. 기본 리포트 선택 시 금액이 6,900원인지 확인
6. 확장 리포트 선택 시 금액이 13,900원인지 확인

### 11.6 결제 차단

1. 테스트 데이터에서 `actionableItemCount`를 0으로 만든다.
2. 결제 화면 진입
3. 경고 문구 표시 확인
4. 데모 결제 비활성 확인
5. 토스 결제 비활성 확인

### 11.7 PDF 저장

1. 상세 리포트 진입
2. `예상 수령액 리포트 PDF로 저장` 클릭
3. 모달에 월급 입력 필드와 2개 버튼만 있는지 확인
4. 월급 없이 저장 클릭
5. `/report/[id]/pdf`로 이동하는지 확인
6. PDF 문서가 화면에 표시되는지 확인
7. `PDF 저장하기` 버튼이 있는지 확인
8. 월급 입력 후 저장 클릭
9. `/report/[id]/pdf?monthlyAverageWage=...`로 이동하는지 확인
10. 계산된 금액이 PDF 문서에 반영되는지 확인

### 11.8 모바일 PDF

1. 모바일 폭 390px 기준으로 PDF 화면 진입
2. `이전으로`, `PDF 저장하기` 버튼이 한 열로 표시되는지 확인
3. PDF 문서가 화면 너비 안에 표시되는지 확인
4. `PDF 저장하기` 클릭 시 브라우저 저장/인쇄 UI가 열리는지 확인

## 13. 향후 개선 과제

### 13.1 플랜 권한 저장

현재는 플랜 선택이 결제 금액에만 반영된다. 운영 서비스에서는 다음 데이터가 필요하다.

- 결제 플랜 ID
