# LIFT 배포 가이드

## 권장 구성

- 백엔드: Render Web Service(Docker)
- 프론트엔드: Netlify Next.js
- 데모 배포: PostgreSQL 영구 DB + mock OAuth

영구 저장이 필요하면 Render Web Service에 PostgreSQL datasource 환경변수를 연결해야 합니다. 연결하지 않으면 Spring Boot가 H2 임시 DB로 기동할 수 있고, Render 재시작/재배포 시 데이터가 사라집니다.

## 1. 백엔드 Render 배포

1. Render에서 새 Web Service를 만듭니다.
2. Root Directory는 프로젝트 루트(`/LIFT`)로 둡니다.
3. Runtime은 Docker를 선택합니다.
4. 환경변수를 설정합니다.

필수:

```text
JWT_SECRET=32바이트_이상의_랜덤_문자열
LIFT_CORS_ALLOWED_ORIGINS=https://프론트엔드-도메인.netlify.app
LIFT_BASE_URL=https://백엔드-도메인.onrender.com
LIFT_FRONTEND_BASE_URL=https://프론트엔드-도메인.netlify.app
OAUTH_MOCK_ENABLED=false
KAKAO_CLIENT_ID=카카오_REST_API_KEY
KAKAO_CLIENT_SECRET=카카오_CLIENT_SECRET_사용시
KAKAO_REDIRECT_URI=https://프론트엔드-도메인.netlify.app/login/callback/kakao
NAVER_CLIENT_ID=네이버_CLIENT_ID
NAVER_CLIENT_SECRET=네이버_CLIENT_SECRET
NAVER_REDIRECT_URI=https://프론트엔드-도메인.netlify.app/login/callback/naver
```

영구 DB:

```text
SPRING_DATASOURCE_URL=jdbc:postgresql://호스트/DB명?sslmode=require
SPRING_DATASOURCE_USERNAME=DB_사용자
SPRING_DATASOURCE_PASSWORD=DB_비밀번호
SPRING_JPA_HIBERNATE_DDL_AUTO=update
```

선택:

```text
AUTH_ACCESS_TOKEN_TTL=1h
AUTH_REFRESH_TOKEN_TTL=14d
GOV24_PUBLIC_SERVICE_ENABLED=false
TOSS_PAYMENTS_ENABLED=false
OPENAI_ENABLED=false
```

Render 배포 후 백엔드 URL 예시는 다음 형태입니다.

```text
https://lift-api.onrender.com
```

## 2. PostgreSQL DB 만들기

무료 데모에는 Neon Postgres를 권장합니다. Neon은 Free plan에서 유휴 상태일 때 compute가 scale-to-zero 되고, 프로젝트별 월 CU-hour allowance가 있습니다. Render Web Service는 그대로 두고 DB만 Neon에 두면 Render 재배포 후에도 데이터가 유지됩니다.

1. Neon에서 새 프로젝트를 만듭니다.
2. Connection string에서 JDBC 형식을 선택하거나 아래 형식으로 바꿉니다.

```text
jdbc:postgresql://HOST/DB_NAME?sslmode=require
```

3. Render `lift-api`의 Environment에 `SPRING_DATASOURCE_URL`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD`를 추가합니다.
4. Render를 재배포합니다.

## 3. 프론트엔드 Netlify 배포

1. Netlify에서 새 사이트를 만듭니다.
2. Git 저장소를 연결합니다.
3. 빌드 설정은 `netlify.toml`을 사용합니다.
4. 환경변수를 설정합니다.

```text
NEXT_PUBLIC_API_BASE_URL=https://백엔드-도메인.onrender.com
NEXT_PUBLIC_TOSS_CLIENT_KEY=
```

배포 후 생성된 Netlify 도메인을 Render의 `LIFT_CORS_ALLOWED_ORIGINS`에 다시 넣고 백엔드를 재배포합니다.

Netlify가 화면에서 직접 묻는 경우 값은 다음처럼 넣으면 됩니다.

```text
Base directory: frontend
Build command: npm run build
Publish directory: frontend/.next
```

## 4. 대안: Render에 프론트엔드도 배포

Netlify도 사용하기 어렵다면 Render Web Service로 프론트엔드를 올릴 수 있습니다.

```text
Root Directory: frontend
Build Command: npm install && npm run build
Start Command: npm run start
Environment Variable:
NEXT_PUBLIC_API_BASE_URL=https://백엔드-도메인.onrender.com
```

이 방식은 Next.js를 일반 Node 서버로 실행합니다. 프론트와 백엔드를 둘 다 Render에 올릴 수 있어 관리가 단순하지만, 무료 플랜에서는 잠든 뒤 첫 접속이 느릴 수 있습니다.

## 5. 로컬 검증 명령

백엔드:

```bash
./gradlew clean build
```

프론트엔드:

```bash
cd frontend
npm install
npm run build
```

## 6. 운영 전환 체크리스트

- `OAUTH_MOCK_ENABLED=false`로 변경
- Kakao/Naver OAuth client id, secret, redirect uri 설정
- 카카오 Redirect URI: `https://프론트엔드-도메인/login/callback/kakao`
- 네이버 Callback URL: `https://프론트엔드-도메인/login/callback/naver`
- 네이버 서비스 URL과 카카오 Web 사이트 도메인에는 프론트엔드 도메인 등록
- 심사용 공개 URL 확인: `/login`, `/terms`, `/privacy`
- PostgreSQL 등 영구 DB 연결
- Toss 실결제 키 연결 시 `TOSS_PAYMENTS_ENABLED=true`
- OpenAI 사용 시 `OPENAI_ENABLED=true`, `OPENAI_API_KEY` 설정

## 7. 네이버/카카오 로그인 심사 메모

- 로그인 버튼 클릭 후 각 제공자의 동의 화면으로 이동해야 합니다.
- 소셜 로그인 사용자는 LIFT의 별도 비밀번호를 입력하지 않습니다.
- 이메일/닉네임 등 동의항목은 실제 서비스 화면과 개인정보처리방침에서 사용 목적을 설명해야 합니다.
- 검수 계정이 로그인 후 온보딩, 약관 동의, 리포트 생성 흐름을 확인할 수 있어야 합니다.
